# Harness Agent Configuration Model

## Overview

Rubber Duck ships the same agents to multiple harnesses (Claude Code, OpenCode, and more to come). Each harness expects its own agent-file frontmatter format. This document defines how a single agent source is authored once and rendered per harness at build time.

## Problem

The prior build stored agent metadata in two incompatible places:

- **OpenCode** frontmatter lived inside each source file (`agents/*.agent.md`), copied verbatim to `dist/opencode/`.
- **Claude** frontmatter was generated by the build script from an embedded table (`CLAUDE_SUBAGENT_SPECS`), which stripped the source frontmatter and injected its own.

Two consequences:

1. **Split metadata.** One agent's identity was spread across a source file and a script table, in two vocabularies. Descriptions and tool/permission grants could drift.
2. **Router special-casing.** The router body was duplicated — a full OpenCode file (`agents/rubber-duck.agent.md`) plus a body-only source (`agents/src/rubber-duck.body.md`) — because it followed a different build path than the subagents.

Adding a harness meant editing the build script's embedded tables, not the agents.

## Decision

Author each agent as a **harness-agnostic body plus a per-agent metadata file with explicit per-harness sections.** The build script becomes a thin renderer: for each harness, render that harness's frontmatter from the metadata, append the shared body, and write to `dist/<harness>/`.

Accepted tradeoff: a small config schema and one renderer per harness, in exchange for adding a harness being a new metadata section per agent + one renderer — never an edit to embedded logic.

### Why per-harness sections, not a canonical capability model

Harness permission models are not 1:1. OpenCode uses a `permission:` block of booleans (`read`, `edit`, `task`, …); Claude uses a `tools:` allowlist. A single canonical capability set would require a lossy, opinionated translation layer. Explicit per-harness sections keep grant fidelity and make each harness's surface auditable. This is a non-negotiable: we do not trade permission fidelity for schema tidiness.

## Target layout

```
agents/<name>/
  body.md      # harness-agnostic prompt (single source of truth)
  meta.json    # shared identity + per-harness frontmatter
```

`meta.json` schema:

```json
{
  "name": "duck-simple",
  "description": "Use for simplicity review to reduce overengineering, indirection, and unnecessary abstractions.",
  "harnesses": {
    "claude":   { "tools": "Read, Glob, Grep, Skill" },
    "opencode": {
      "mode": "all",
      "color": "#FFD801",
      "permission": { "read": "allow", "edit": "allow", "skill": "allow" }
    }
  }
}
```

- `name` / `description` are shared identity, rendered into every harness.
- `harnesses.<harness>` holds only that harness's frontmatter fields, rendered verbatim into its format.
- Fields that happen to coincide across harnesses (e.g. `color`) stay per-harness — coincidental overlap is not shared semantics.

The router is not special: it is an agent with a `body.md` and a `meta.json` like any other. Its Claude frontmatter (e.g. `initialPrompt`, agent tools) is just its `harnesses.claude` section.

## Build model

- The builder (`scripts/build-harness-artifacts.sh`) reads each agent's `meta.json` + `body.md` and calls a per-harness renderer (`render_claude_fm`, `render_opencode_fm`, …).
- Rendered artifacts are committed under `dist/<harness>/`. `--check` mode asserts the committed output matches a fresh render, so drift is a CI failure.

### Dependency boundary

The builder is a **build-time / CI tool**, not part of the install path. End users receive pre-built `dist/` artifacts via `scripts/rubber-duck.sh`; they never run the builder. The builder may therefore depend on `jq` for JSON parsing without affecting the minimal `curl | bash` install path. `jq` is added to the builder's `require_cmd` checks.

## Adding a new harness

1. Add a `harnesses.<new>` section to each agent's `meta.json`.
2. Add a `render_<new>_fm` renderer and a `dist/<new>/` output path in the builder.
3. Add the harness's install target to `scripts/rubber-duck.sh` / `rubber-duck.ps1`.

No change to agent bodies or to other harnesses' sections.
